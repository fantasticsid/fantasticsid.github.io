Positioning the traffic lights of your Cocoa App

July 4, 2018

The other day I wanted to find a generic way to reposition the standard buttons (aka traffic lights) of a Mac app. I remember iTunes used to position the traffic lights vertically and I thought it was pretty cool.

Since Mac Yosemite, Mac apps have more choices for the look and feel of the toolbar, as well as the option to fully extend content view to the whole Window. A lot of apps have adopted this style to stand out (Bear, Things 3).

In the app that I have been working on, we relied on setting the toolbar height so that the traffic lights can be shifted down accordingly. Unfortunately that also means we need to reset the toolbar height every time the Window resizes. I wanted to see if there’s a way we can arbitrarily position these buttons, and preferably with auto layout.

Setting frames for traffic lights

In Apple’s documentation, there’re APIs on NSWindow to get the widget instances of these standard buttons.

func standardWindowButton(_ b: NSWindow.ButtonType) -> NSButton?

This is kind of nice as it gives you an NSButton. And indeed by setting the frames of these buttons, I was able to get some interesting results:

  override func windowDidLoad() {
    super.windowDidLoad()

    self.window?.titleVisibility = NSWindow.TitleVisibility.hidden
    self.window?.styleMask.formUnion(NSWindow.StyleMask.fullSizeContentView)
    self.window!.titlebarAppearsTransparent = true

    if let close = self.window?.standardWindowButton(NSWindow.ButtonType.closeButton) {
      let frame = close.frame
      close.frame = NSRect(origin: CGPoint(x: frame.origin.x, y: frame.origin.y - 10), size: frame.size)
    }
  }

(Yep, we pushed the close button down, outside the bounds of the title bar and part of it is cut off)

Unfortunately as I mentioned at the start, as soon as you resize the window, or exit from fullscreen mode, the toolbar seems to put the buttons back in place. So to make this work, you really need to fight the toolbar and reset the frame every time the app quits fullscreen mode, or is resized.

Using auto layout with traffic lights

Can we do better? Let’s see what the toolbar is doing to reposition the buttons. Xcode lets you inspect view hierarchy of your App at runtime, and also lists all constraints associated with the views. That’s pretty neat!

Looks like these are constraints associated with the close button widget. So If I change these constraints, would that keep the close button fixed at the position we want?

I experimented with mutating the constraints, but unfortunately this approach did not quite work. These constraints do seem to be mutable, but they just don’t seem to affect how the buttons are repositioned. I’m pretty sure there’re more than meets the eye when the toolbar repositions these buttons, but I wasn’t able to find any more details from inspecting the view hierarchy at runtime.

So it seemed that the toolbar has its own way of dictating the constraints, and this is understandable since the toolbar/titlebar has a lot more cases to deal with.

But what if we take the buttons out of the titlebar, and put them inside a root level view that we can control?

  var buttons: [NSView] = []
  override func windowDidLoad() {
    super.windowDidLoad()

    self.window?.titleVisibility = NSWindow.TitleVisibility.hidden
    self.window?.styleMask.formUnion(NSWindow.StyleMask.fullSizeContentView)

    self.window!.titlebarAppearsTransparent = true

    let close = self.window?.standardWindowButton(NSWindow.ButtonType.closeButton)
    let minimize = self.window?.standardWindowButton(NSWindow.ButtonType.miniaturizeButton)
    let maximize = self.window?.standardWindowButton(NSWindow.ButtonType.zoomButton)

    buttons.append(close!)
    buttons.append(minimize!)
    buttons.append(maximize!)

    var offsetX = 10
    let contentView = self.window!.contentView!

    buttons.forEach { (btn) in
      btn.superview?.willRemoveSubview(btn)
      btn.removeFromSuperview()

      btn.viewWillMove(toSuperview: contentView)
      contentView.addSubview(btn)
      btn.viewDidMoveToSuperview()
    }

    buttons.forEach { (btn) in
      btn.snp.makeConstraints({ (make) in
        make.left.equalTo(contentView.snp.left).offset(50)
        make.top.equalTo(contentView.snp.top).offset(offsetX)
        make.height.equalTo(14)
        make.width.equalTo(14)
      })
      offsetX += 22
    }
    contentView.layoutSubtreeIfNeeded()
  }

Basically for the purpose of experiments, we took these buttons out of their view hierarchy, and then added it to the root level content view.

And it worked!

We are also using auto layout to manage the positioning. If we resize now, the buttons keep where they are.

At this point, we can position the traffic lights anywhere in our window, and arrange them in any way we like! Great!

However, as soon as I started interacting with them, I saw this:

The buttons are positioned where we want them, but when I mouse over, the area that’s detecting the mouse movement is wrong - they stayed at the old spot. That’s why in the screenshot, even though my mouse is not over the traffic lights, the buttons still render the small icons inside as if they are “activated”. As soon as I resized the window however, the areas are recalculated and everything is perfect. So the question now becomes:

  • Can we update these tracking areas?
  • How are they updated when the window is resized?

Attempting to update tracking areas

I know that in a regular cocoa app, you’d use NSTrackingArea to subscribe to mouse move events in a certain area. So let’s see if we can modify the tracking area manually:


    let timer = Timer(timeInterval: 1, target: self, selector: #selector(self.updateTrafficTrackingAreas), userInfo: nil, repeats: true)

    RunLoop.main.add(timer, forMode: .defaultRunLoopMode)


  @objc func updateTrafficTrackingAreas() {
    let btn = self.window?.standardWindowButton(NSWindow.ButtonType.closeButton)
  }

I created a timer that would fire once per second, and put a breakpoint on the callback.

So looks like the NSThemeFrame, which is the top level view in the NSWindow, has 9 tracking areas. Sorting them by size, we get:

Origin (-4.0, -4.0), Size (16.0, 16.0)
Origin (-4.0, 258.0), Size (16.0, 16.0)
Origin (468.0, 258.0), Size (16.0, 16.0)
Origin (465.0, -4.0), Size (19.0, 19.0)

Origin (-4.0, 12.0), Size (7.0, 246.0)
Origin (12.0, 267.0), Size (456.0, 7.0)
Origin (477.0, 15.0), Size (7.0, 243.0)
Origin (12.0, -4.0), Size (453.0, 7.0)

Origin (7.0, 251.0), Size (54.0, 16.0)

My guess, is that the first four of them are used for detecting when mouse is over the 4 corners of the window, so that the NSThemeFrame shows a resize handle. And same for the next four tracking areas, they are used for detecting mouse movement around the 4 edges. And I believe the last one is for detecting mouse movement around the traffic lights, since the size corresponds to a horizontal rectangle. Unfortunately, I could not get Xcode to mutate the tracking areas, since they are getter only properties. Plus, it would be hard to compute an accurate area that would match our new button positions.

Asking AppKit to update tracking areas

So it seems our option now is to look into how resizing a window triggers the tracking area update. I know that to add a tracking area, the [NSView addTrackingArea:] API is used. And it turns out that Xcode let us create symbolic breakpoints! (Debug -> Breakpoints -> Create symbolic breakpoints)

Once I added the breakpoint, restarted the app and started resizing, Xcode suspended the App! That means our symbolic breakpoint is working!

Of course AppKit does not come with source so all you can see is the assembly. But Xcode was still able to show you the stack trace and symbolize every function call on the stack:

That’s really awesome! Looking at the stack trace, the NSThemeFrame is indeed handling the resize event, and it was calling [NSThemeFrame viewDidEndLiveResize] which in turn calls addTrackingArea several levels down. I’m assuming all APIs that starts with an underscore are somewhat private, but viewDidEndLiveResize seems a legit API that we can call into! All I need to do is:

override func windowDidLoad() {
...
    let contentView = self.window!.contentView!
// move buttons into content view, add auto layout constraints
    contentView.layoutSubtreeIfNeeded()
    contentView.superview!.viewDidEndLiveResize()
...
...
}

And it worked!!! As soon as the window appeared, the buttons had the correct tracking areas set up and would activate when they are moused over!

Conclusion

At this point, we have mostly achieved what we wanted to do: We can position the traffic lights any way we like, with the correct tracking areas set up, and we did that without using any private APIs! Moreover, since we are using auto layout within content view, we can achieve fancier effects by setting up constraints between our app specific views and these traffic lights! (I haven’t experimented with this idea yet though)

I should also say that this definitely is not without downsides. Firstly, I doubt this is how AppKit expects apps to manipulate its traffic lights, as we’ve seen it’s hard to adjust these buttons and work with the constraints the toolbar added. Secondly, we haven’t handled some cases here: exiting from fullscreen puts these buttons back, but that should be easy to fix. Lastly, since we moved the traffic lights into the content view, it’s now our responsibilities to make sure they are always visible and interactive.

That said, working with Xcode and trying to guess what AppKit is doing at runtime has been a great experience!
